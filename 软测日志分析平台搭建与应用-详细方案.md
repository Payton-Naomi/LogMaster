# 课题九：软测日志分析平台搭建与应用 — 详细方案

## 一、项目概述

### 1.1 背景与目标

嵌入式行车记录仪在研发和测试过程中，通过串口（UART）输出大量运行日志。目前日志散落在各台测试电脑上，依赖人工逐行排查，效率低、易遗漏。本方案旨在搭建一套企业级统一日志管理平台，实现多产线日志的自动采集、集中存储、智能解析与可视化展示。

核心目标：让设备日志「说话」——从被动查看到主动告警，从人工排查到 AI 辅助诊断。

### 1.2 约束条件

| 约束项 | 说明 |
|--------|------|
| 团队规模 | 3 人，均为测试工程师，具备一定开发基础 |
| 时间预算 | 6 周，工作日每天 1~2 小时（总计约 45~60 小时/人） |
| 预算上限 | 8000 元（含云服务器、大模型 API 费用） |
| 目标设备 | 行车记录仪（嵌入式 Linux / RTOS），串口日志输出 |
| 落地要求 | 真实可用的企业级项目，非 Demo 原型 |

## 三、技术选型

### 3.1 后端语言：Go（推荐）

| 维度 | Go | Python | Java |
|------|-----|--------|------|
| 并发模型 | Goroutine (2KB/协程) | asyncio/GIL 限制 | Thread (1MB/线程) |
| 部署复杂度 | 单二进制文件 | 需运行时环境 | 需 JVM |
| 内存占用 | 低（~10MB 起步） | 中等 | 高 |
| 串口库成熟度 | `go.bug.st/serial` 成熟 | `pyserial` 最成熟 | `jSerialComm` 成熟 |
| 学习曲线 | 团队需学习 | 团队可能更熟悉 | 团队可能熟悉 |
| 高并发吞吐 | 优秀 | 一般 | 优秀但重量级 |

**结论**：Go 的 Goroutine 模型天然适合多设备并发采集场景。每个设备一个 Goroutine，30 台设备内存仅增加约 60KB。编译为单二进制文件，挂测 PC 上部署零依赖。团队三人有一定开发基础，Go 语法简洁，学习成本可控。

**Go 技术栈选型**：

| 组件 | 选型 | 理由 |
|------|------|------|
| Web 框架 | **Gin** | 高性能、文档丰富、社区活跃、学习曲线平缓 |
| 串口通信 | **go.bug.st/serial** | 跨平台、API 简洁、维护活跃 |
| 数据库 ORM | **GORM** | Go 生态标准 ORM，支持 PostgreSQL |
| HTTP 客户端 | **Resty** | 链式调用，Agent 上传日志简洁可靠 |
| 任务调度 | **robfig/cron** | Go 标准 cron 库，支持秒级精度 |
| 配置管理 | **Viper** | 支持 YAML/JSON/环境变量 |
| 日志库 | **Zap** | 高性能结构化日志 |

### 3.2 前端框架：Vue 3

| 组件 | 选型 | 理由 |
|------|------|------|
| 核心框架 | **Vue 3 + Vite** | 学习曲线友好，适合后转全栈团队 |

### 3.3 数据存储

| 存储 | 选型 | 用途 |
|------|------|------|
| 业务数据库 | **PostgreSQL 15+** | 设备管理、用户、解析规则、任务记录、告警配置 |
| 日志存储 | **Grafana Loki** | 日志正文全文检索、标签索引、时序存储 |
| 日志可视化 | **Grafana** | 对接 Loki 的可视化仪表板 |

**选型理由**：Loki 相比 ELK（Elasticsearch）存储成本低 80% 以上，只索引标签不索引日志正文，压缩存储。对于 3 人团队，运维成本极低。Grafana 可同时对接 Loki 和 PostgreSQL，一个界面覆盖日志检索和业务数据展示。

### 3.4 AI 大模型：双轨策略

挂测 PC 无独立显卡，本地模型只能纯 CPU 推理，需重新评估可行模型。

**CPU-only 推理实测数据**（Q4 量化，i5/Ryzen 5 级别 CPU）：

| 模型 | 参数 | 内存占用 | CPU 推理速度 | 日志分析适用性 |
|------|------|----------|-------------|---------------|
| Qwen3 0.6B | 0.6B | ~1.5GB | 30~50 tok/s | 异常检测专项，已知 92% 准确率 |
| Qwen3 1.8B | 1.8B | ~3GB | 15~25 tok/s | 轻量可用，分类+简单推理 |
| Qwen3 7B (Q4_K_M) | 7B | ~5GB | 5~10 tok/s | 平衡点，日常分析主力 |
| Qwen3 14B | 14B | ~9GB | 2~4 tok/s | 太慢，不推荐 CPU-only |

**结论**：14B 在纯 CPU 下推理速度仅 2~4 tok/s，分析一批日志需数分钟，不适合生产使用。CPU-only 环境下，本地模型的天花板是 **Qwen3 7B + Q4_K_M 量化**，推荐**三层模型矩阵**：

| 层级 | 模型 | 用途 | 推理速度 | 部署方式 |
|------|------|------|----------|----------|
| 第 1 层：规则引擎 | ripgrep 关键字匹配 | 毫秒级扫描，覆盖 95% 已知异常 | 毫秒级 | 本地 |
| 第 2 层：小型本地模型 | Ollama + Qwen3 0.6B/1.8B | 快速异常分类、级别判定 | 1~3 秒/批 | 本地 CPU |
| 第 3 层：云端大模型 | 通义千问 Qwen3-Max | 深度根因分析、跨文件因果推断 | 3~5 秒/批 | 云端 API |

**具体策略**：

- 关键字命中后，Qwen3 0.6B 做初步异常分类（已知在日志场景准确率 92%，单条仅 11ms）
- 0.6B 无法判断的复杂模式，扔给云端 Qwen3-Max 深度推理
- 7B 作为本地备选，当需要深度推理但网络受限时启用（5~10 tok/s 可接受，毕竟日志分析非实时场景）
- 数据不出域时用 7B 本地推理，正常情况用云端 API

**硬件要求**：普通 PC（16GB 内存，CPU i5/Ryzen 5 以上，无 GPU）即可运行 Qwen3 7B（Q4 量化），推理延迟约 5~10 tok/s，对日志分析场景完全可接受。0.6B 模型 8GB 内存即可运行。

## 五、核心模块设计

### 5.1 串口日志采集 Agent（Go）

**功能**：部署在挂测 PC 上，定时/实时采集行车记录仪串口日志并上传至后端。

**核心流程**：

```
启动 → 读取配置文件 → 遍历设备列表 → 
为每个设备启动 Goroutine → 
  打开串口 → 循环读取日志行 → 
  写入本地缓冲文件 → 
  定时触发(如每 5 分钟) → 
    压缩日志 → HTTP POST 上传至后端 → 
    清理本地缓冲
```

**关键设计点**：

- **断线重连**：串口异常断开时自动重试，指数退避（1s→2s→4s→max 30s）
- **本地缓冲**：网络异常时日志暂存本地 SQLite，恢复后补传，确保不丢数据
- **心跳上报**：每 30 秒上报 Agent 状态（设备在线数、采集速率）
- **配置热加载**：支持通过后端 API 远程更新设备配置（波特率、日志级别过滤）
- **资源控制**：单 PC 最多支持 20 路串口并发采集，内存占用 < 100MB

### 5.2 Web 后端 API（Go + Gin）

**模块划分**：

| 模块 | 核心 API | 说明 |
|------|----------|------|
| 日志接收 | `POST /api/v1/logs/upload` | 接收 Agent 上传的日志文件，写入 Loki |
| 设备管理 | CRUD `/api/v1/devices` | 设备注册、状态管理 |
| 项目管理 | CRUD `/api/v1/projects` | 多产线/多项目管理 |
| 解析规则 | CRUD `/api/v1/rules` | 规则配置、关键字管理 |
| 解析任务 | `POST /api/v1/tasks` | 创建解析任务，触发关键词匹配 + AI 分析 |
| 告警查询 | `GET /api/v1/alerts` | 告警列表查询 |
| AI 分析 | `POST /api/v1/ai/analyze` | 异步调用大模型分析日志 |
| 统计报表 | `GET /api/v1/stats/*` | 日志量统计、异常趋势等 |

**架构分层**：

```
Handler (路由/参数校验) → Service (业务逻辑) → Repository (数据访问) → DB
                                    │
                                    ├── Loki Client (日志检索)
                                    └── AI Client (大模型调用)
```

### 5.4 日志解析引擎

解析采用**关键词快速扫描 + 上下文 AI 分析**的双层架构，兼顾速度和深度：

#### 第一层：关键字快速搜索（速度优先）

本质是利用系统级文本搜索工具批量扫描，秒级出结果，不需要逐行加载全量日志：

**PowerShell（Windows）**：
```powershell
Get-ChildItem "C:\日志目录" -Recurse -Filter "*.log" |
  Select-String -Pattern "FAT-fs|mmc0: error|Buffer I/O|mmc0: Timeout" |
  Format-Table Filename, LineNumber, Line
```

**Linux/Mac**：推荐使用 `ripgrep`，比原生 `grep` 快 5~10 倍：
```bash
rg "FAT-fs|mmc0: error|Buffer I/O" /日志目录/ --glob="*.log"
```

多关键字并行搜索：把多个关键字拼成正则，一次扫描覆盖。在已上传的日志会话中，Loki 基于标签过滤后，后端调用系统级搜索工具完成快速匹配。

#### 第二层：时间过滤

关键字命中后，从匹配行提取时间戳进行二次过滤：

```python
import re
from datetime import datetime
cutoff = datetime(2026, 6, 1, 14, 30, 0)
timestamp_pattern = r'\[(\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2})'
def filter_by_time(matched_lines):
    for line in matched_lines:
        ts = re.search(timestamp_pattern, line)
        if ts and datetime.strptime(ts.group(1), "%Y-%m-%d %H:%M:%S") >= cutoff:
            yield line
```

#### 第三层：上下文提取与多文件对比

关键字命中后，提取命中行前后 N 行上下文供 AI 分析：

```python
def extract_context(file_path, line_number, before=30, after=30):
    with open(file_path, 'r') as f:
        lines = f.readlines()
    start = max(0, line_number - before - 1)
    end = min(len(lines), line_number + after)
    return ''.join(lines[start:end])
```

把同一关键字在不同文件、不同时间点的上下文全部提取出来，进行横向对比：
- 相同模式 → 识别共性根因
- 差异变化 → 判断问题严重程度演化
- 时序关联 → 推断因果链

**预设关键字规则**（行车记录仪领域）：

| 模块 | 关键字 | 级别 | 含义 |
|------|--------|------|------|
| storage | "write error", "sdcard full", "mount fail", "FAT-fs" | error | 存储异常 |
| camera | "sensor init fail", "frame drop", "isp timeout" | error | 摄像头异常 |
| gps | "no fix", "signal lost", "gps timeout" | warn | GPS 信号丢失 |
| system | "watchdog reset", "oom", "panic", "assert" | fatal | 系统致命错误 |
| system | "battery low", "temp high", "voltage drop" | warn | 电源/温度告警 |
| network | "wifi disconnect", "4g no service" | warn | 网络异常 |
| storage | "loop recording start", "file saved" | info | 正常录像 |

#### 第四层：AI 语义分析（深度理解）

规则引擎筛选出的异常日志，结合上下文批量提交给大模型进行语义分析：

- 跨日志行的因果推断（如 "sensor init fail" → "camera module crash"）
- 未知错误模式的自动识别和归类
- 生成可读的诊断报告和修复建议

### 5.5 AI 分析引擎

AI 分析分为**云端 API** 和**本地模型**两种部署模式，可根据网络环境和成本灵活选择：

#### 云端模式：通义千问 API（推荐生产环境）

**完整调用流程**：

```

  多文件横向对比去重 → 批处理(每批≤50条) → 
  构造 Prompt → 调用通义千问 API → 
  解析 JSON 结果 → 写入 parse_results + alerts
```

#### 本地模式：三层级本地策略（适配 CPU-only，无 GPU）

- 零 API 费用，一次部署持续使用
- 数据不出本地，满足隐私合规要求
- Qwen3 0.6B 在 8GB 内存即可运行，0.6B/1.8B 对低配电脑非常友好

#### 混合模式（推荐，适配 CPU-only）

```
┌─────────────────────────────────────────────────────┐
│              AI 分析调度策略（无独立显卡）              │
├─────────────────────────────────────────────────────┤
│  1. 关键字扫描 → 毫秒级筛选命中点                     │
│  2. Qwen3 0.6B 本地 → 快速异常分类（1~3秒出结果）       │

│  4. 复杂不确定模式 → 云端通义千问 Qwen3-Max → 深度推理  │
```

**成本控制**：

- 非关键日志（info/debug）不送入 AI，仅规则引擎处理
- 关键字扫描命中的异常，先用 Qwen3 0.6B 本地做初步分类（零成本）
- 仅复杂/不确定的异常模式才调用云端 API（约占总异常量的 5%~10%）

- 估算每月云端 API 费用约 50~150 元，远低于全量 API 分析方案

### 5.6 离线日志快速分析（独立工具链）

当无需完整平台时，一套轻量级命令行工具可以在**分钟内**完成日志分析。此方案也可作为平台"离线诊断"功能的底层实现。

**工具链**：

| 组件 | 推荐方案 | 用途 |
|------|----------|------|
| 文本搜索 | `ripgrep`（跨平台）或 PowerShell `Select-String` | 关键字快速扫描 |
| 时间解析 | Python `re` + `datetime` | 时间戳过滤 |
| 上下文提取 | Python 脚本 | 命中行前后 N 行提取 |
| AI 推理 | Ollama + Qwen 0.6B/7B | 根因分析、模式识别 |
| 前端交互 | 命令行 / Streamlit | 输入目录和关键字 |

**最小可行方案（MVP）**：

```
用户输入目录 + 关键字
  → ripgrep / Select-String 扫描
  → Python 过滤时间戳
  → 提取上下文拼接成 prompt
  → Ollama 本地推理输出分析结果
```

**Windows 一键脚本**：
```powershell
# search.ps1 — 关键字搜索 + 结果导出
param($Path, $Pattern)
Get-ChildItem $Path -Recurse -Filter "*.log" |
  Select-String -Pattern $Pattern |
  Select-Object Filename, LineNumber, Line |
  Export-Csv "search_result.csv" -NoTypeInformation
```

**Python 分析脚本**：
```python
# analyze.py — 上下文提取 + Ollama 推理
import subprocess, re, ollama
from datetime import datetime

def search_logs(directory, pattern):
    result = subprocess.run(
        ["rg", pattern, directory, "--glob=*.log", "-n", "--no-heading"],
        capture_output=True, text=True
    )
    return result.stdout.strip().split("\n")

def filter_by_time(lines, cutoff):
    for line in lines:
        ts = re.search(r'\[(\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2})', line)
        if ts and datetime.strptime(ts.group(1), "%Y-%m-%d %H:%M:%S") >= cutoff:
            yield line

def ai_analyze(context):
    prompt = f"""分析以下行车记录仪日志上下文，回答：
1. 错误发生前后的关键事件
2. 不同文件之间的共性和差异
3. 从系统层面判断根因

日志：{context}"""
    return ollama.chat(model="qwen3:0.6b", messages=[{"role":"user","content":prompt}])
```

**使用场景**：挂测 PC 本地快速排查（20 秒出结果），日志文件批量扫描（Windows/Linux 均可），无需启动 Web 平台，适合应急排障。

| 4 | 用户管理/登录 | 内网使用可以不登录 |
| 5（最先砍） | 数据导出 | 手动从数据库导出也行 |

## 八、风险与应对策略

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 串口通信不稳定 | 高 | 中 | 断线重连 + 本地缓冲 + 补传机制 |
| Go 学习曲线超预期 | 中 | 高 | 第 1 周集中学习 Go 基础；核心逻辑可先用 Python 验证再翻译 |
| 大模型 API 调用超预算 | 低 | 中 | 严格去重 + 批处理 + 切换为更便宜的 Qwen-Plus |
| Loki 部署/运维困难 | 中 | 中 | 准备降级方案：日志直接存文件 + PostgreSQL |
| 时间不足 | 中 | 高 | 按优先级裁剪：先保证采集→存储→检索→解析核心链路 |
| 行车记录仪设备不足 | 中 | 低 | 可用模拟器生成日志数据先行开发 |

**降级策略**：如果时间紧张，AI 分析模块可简化为直接调用 API 的 Prompt 模式，不做复杂的 Prompt 工程和多轮对话。
