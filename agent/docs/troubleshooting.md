# 故障排查

## 客户端无法启动

确认 EXE 为 Windows amd64 构建，检查 WebView2 Runtime 是否安装，并查看 Windows 事件查看器和 `collector.stderr.log`。若提示 WebView2 缺失，安装 Evergreen Runtime 后重启；不要从不受信任来源下载运行库。

## 串口不可见或无法连接

关闭占用串口的调试工具，核对设备管理器中的 COM 号、驱动、VID/PID 和 USB 序列号。拔插后 COM 号变化时优先使用稳定硬件标识。单路故障时只断开并重连该路，确认其他三路仍持续增长。

## 有连接但没有日志

核对波特率、数据位、停止位、校验位、流控、编码和行结束符。检查设备是否实际输出数据，以及日志目录最新文件时间。不要通过不断重启全部设备掩盖单路配置错误。

## 上传积压

检查后端 `/api/health`、代理超时、TLS、上传鉴权和系统时间。`pending` 可等待重试；`uncertain` 必须先用文件 SHA-256、项目、版本和时间在平台核对。未核对前不要自动重传，以免后端生成重复任务。

## 重启后队列异常

保留 SQLite、spool 和日志目录副本，检查是否有权限错误、磁盘满或外部程序锁定文件。不要手工编辑数据库。恢复前记录 `pending/uploading/uncertain/dead` 数量和相关文件 SHA-256。

## 磁盘空间不足

停止新增采集，保留所有未上传文件。只能按运维审批清理已确认上传且超过保留期的数据。配置阈值不是操作系统配额，现场仍需磁盘监控告警。

## 长时测试失败

保留 `events.ndjson`、`samples.ndjson`、四个模拟器 NDJSON、标准输出、错误输出和报告。以故障发生时间关联客户端日志；检查单路序号是否停滞、内存是否持续单调增长、队列是否恢复及上传是否重复。
